## Plan: Unified Sync System Redesign

Replace the current hybrid sync architecture with one coherent model: repo-authored entity definitions are the only authoritative source; build/publish turns them into runtime state; preview compiles a fully explicit executable plan; execute only runs that plan; policy, configuration, and explainability are part of the same contract rather than bolted on through fallbacks or boot-time globals. The recommended approach is to collapse the current JSON/YAML/SQLite/fallback chain into one declarative definition model with strict compilation, versioned publication, and auditable decision logs from preview through execute.

**Steps**
1. Phase 0 — Freeze the architectural principles and scope. Adopt these non-negotiable rules as the redesign contract: one authoritative definition source in-repo; no runtime fallback to alternate config sources; no entity-type branching that changes business flow outside compiled definitions; no generated artifact treated as a peer configuration source; preview must fully resolve flow/policy/scope into the saved plan; execute must be a generic plan runner; audit must capture not just events but decisions and policy evaluations. This phase blocks all lower-level design work because every later step depends on these boundaries.
2. Phase 1 — Define the canonical domain model for the whole system. Replace the current partial `SyncRecipe` mental model with a first-class definition set that covers: metadata scope, execution graph, post-metadata steps, governance policies, allowed schemas, environment/service bindings by reference, explainability metadata, and version/provenance. The key output is a typed contract for `EntitySyncDefinition`, `CompiledSyncPlan`, `ExecutionStep`, `GovernancePolicyRef`, and `DecisionRecord`. This phase should deliberately separate authoring-time declarations from compiled runtime artifacts so the system is understandable at a glance.
3. Phase 2 — Make repo-authored definitions the single source of truth. Introduce one authoritative definitions directory in the repo and retire the current multi-source chain where `deploy/mssql/sync-recipes.json` feeds derived YAML and runtime still falls back between registry and bundled JSON. Definitions should be human-reviewed, versioned, and directly expressive enough that generation scripts are optional tooling, not part of runtime correctness. Publication should compile validated definitions into runtime storage, but runtime must never infer missing behavior from alternate sources.
4. Phase 3 — Introduce an explicit compile/publish boundary. Add a compile step that reads authoritative definitions and emits a versioned runtime package for the server/runtime to consume. The compiler should resolve table scope, execution order, allowed schemas, governance references, and all execution steps ahead of time; reject incomplete entities; stamp provenance; and publish the compiled result into runtime storage atomically. The server bootstrap should only load published runtime definitions and should fail loudly on missing or invalid published data rather than silently falling back.
5. Phase 4 — Redesign preview as a pure compiler from published definition to executable plan. `previewSync` should stop deciding behavior through mixed resolver/fallback logic and instead: load the already-published entity definition, resolve environment bindings, evaluate optional inputs, compile the exact `executeFlow.steps`, evaluate governance and risk inputs, and persist a plan that is already complete. The plan must carry everything execute needs: flow steps, allowed schemas, scope decisions, policy references and evaluated gate context, provenance/version, and explainability records. No later stage should recompute business flow from `entityType`.
6. Phase 5 — Redesign execute as a generic plan runner. `executeSync` should enforce runtime safety rails and then iterate through the snapshotted flow. Shared engine logic such as transaction handling, telemetry, retries, and failure semantics stays in the executor; entity-specific business flow must live only in the compiled plan. Contract becomes a peer entity with more steps, not a privileged code path. This phase specifically removes the special structural role of `execute-pipeline.ts`, the contract-only pre-transaction branch in `execute.ts`, and `resolvePostMetadataActions(...)` as a business-flow fallback.
7. Phase 6 — Unify governance and environment configuration into the same model. Freeze windows, approval policy, risk multipliers, allowed target roles, and service endpoints should all be resolved through the same published definition/config package rather than through module-global registries, partial environment files, or code-derived defaults. Governance evaluation must be explicit, refreshable, versioned, and recorded per run. Environment/service configuration should remain externalized but referenced declaratively from the published plan so a run is reproducible and auditable.
8. Phase 7 — Make explainability a first-class artifact. Extend the plan and event model so the system records not only what happened but why: definition version used, scope decisions, optional-table inclusion rationale, policy evaluations, risk gates, compile-time warnings, and per-step semantic purpose. The UI and API surfaces should consume these same records rather than reverse-engineering intent from raw events. The operator should be able to inspect a run and answer: why this entity used this flow, why this table was included, why execute was permitted or blocked, and what exact definition version was applied.
9. Phase 8 — Simplify operational surfaces around the new contract. Replace the current conceptual split between recipe bundle, registry entity projection, event reconstruction, and UI-specific interpretation with a stable pipeline: authoritative definition -> compiled published definition -> compiled executable plan -> execution events and decision log -> UI/API rendering. Any generated artifacts kept for compatibility should be explicitly marked as derived outputs with no runtime authority. This phase should also retire ambiguous naming like “recipe” where it obscures whether something is authored, compiled, or executed.
10. Phase 9 — Migrate incrementally without mixed-end-state ambiguity. Move one entity at a time onto the new definition model behind a compile-time compatibility boundary, but do not allow permanent dual-authority behavior. During migration, old entities may remain on the legacy path, while migrated entities must be definition-only and fallback-free. Start with the smallest peer entities that prove the model cleanly, then move dataset/rule, and fold contract into the same structure only after the generic step model is proven.
11. Phase 10 — Verification and acceptance. Add validation at four levels: definition-schema validation, compiler tests, plan/executor tests, and end-to-end operator explainability tests. Final acceptance requires proving that a reviewer can trace any run from authored definition through compiled plan to execution and UI evidence without consulting hidden fallback logic or scattered configuration files.

**Decisions**
- Included: runtime architecture, configuration model, publication/bootstrap model, governance, explainability/audit contract, and migration shape for the sync system as one coherent redesign.
- Excluded: unrelated diff-algorithm changes, unrelated UI styling work, and speculative rewrites of target external systems beyond the boundaries required to express and execute explicit steps.
- Decision: the authoritative source is repo-authored declarative definitions published into runtime storage; DB/runtime state is a deployed artifact, not the authoring truth.
- Decision: explainability is part of the architecture, not a later observability enhancement.
- Decision: generated JSON/YAML artifacts may remain temporarily for migration/export, but in the target state they are derived outputs with zero runtime authority unless they are the single published runtime package.
- Recommendation: rename concepts so authored definition, compiled published definition, executable plan, and run history are distinct nouns. The current overloaded “recipe” term hides too much.
- Recommendation: do not preserve long-term hybrid resolution. During migration, allow compatibility only at the entity cutover boundary, never as an always-on resolver fallback inside a single entity path.
- Clarification: this redesign is not frontend-first. The primary work is correcting preview/plan/execute architecture and the authoritative-definition model so the backend contract is right. Existing UI surfaces that already work well, especially `EnvSync`, should be preserved and re-pointed to cleaner backend contracts rather than redesigned for their own sake.
- Clarification: `SyncAdmin` should be strengthened carefully, not bloated. Existing sections/components are a good base and should be reused where they align with the operational model. Proposal/approval/evidence capabilities should only move into `SyncAdmin` in a way that simplifies the operator mental model and reflects real backend workflow; avoid adding parallel complexity or packing unrelated controls into one screen.
- Clarification: existing BE and FE work across `EntityRegistry`, `SyncAdmin`, `SyncProposals`, `SyncApprovals`, and `SyncEvidence` should be treated as reusable assets and migration inputs for the architecture redesign. The backend plan must account for what is already wired so the new preview/execute/plan model lands on top of real capability instead of discarding it.


- Clarification: the strict repo-authority rule applies to structural sync semantics and topology that must be reproducible and deterministic: entity definitions, execution steps, schema scope, SCD2 strategies that affect runtime behavior, and environment/service topology. Time-varying operational governance such as approval requests, freeze-window instances, schedules, routes, and evidence records can remain runtime-managed if each has one explicit authoritative store, versioning where needed, and no shadow file fallback.

**Further Considerations**
1. Recommended source model: one file per entity definition plus shared schema for reusable step templates and governance references, rather than one giant aggregate file. This keeps review, ownership, and diffs local.
2. Recommended publication model: compile all definitions in CI and publish a versioned runtime package; server startup should load that package or fail, rather than silently seed partial runtime state.
3. Recommended migration order: prove the model on pipelineActivity or gateMetadata first, then dataset/rule, then contract normalization once the generic step executor and explainability contract are stable.
6. Recommended widget model: keep `EnvSync` unchanged as the single daily-work surface; keep `EntityRegistry` separate as the structural-definition workspace; keep one `SyncAdmin` console for the rest of sync operations; move proposal, approval, and evidence responsibilities behind `SyncAdmin` sections or retire them if their workflow is not part of the core mission. Do not keep them as separate peer dashboard widgets because that fragments the operator’s mental model.
7. Recommended prioritization test for every sync widget/section: if it is needed to define sync structure, govern sync operations, or prove/audit a sync run, it belongs either in `EntityRegistry` or `SyncAdmin`; if it is merely a parallel dashboard view of the same workflow, collapse it.

4. Recommended admin operating model: admins can still create entities, assign SCD2 strategies, and manage governance, but not by silently mutating runtime authority in place. The clean model is draft/propose/publish: admin UI creates or edits a draft definition, the system validates it, publishes it into runtime storage, and records provenance. In stricter environments the publish step can require a repo-backed PR/merge; in more operational environments the UI can generate the equivalent commit/package and publish it directly, but the resulting published artifact remains the only runtime authority.
5. Recommended widget role split: Entity Registry should evolve from an authoritative live registry editor into a definition catalog plus draft/publish surface; Sync Admin remains for operational policy and environment governance; Sync Proposals, Sync Approvals, Sync Evidence, and the main Env Sync widget remain downstream workflow/execution surfaces and largely survive, but they should read explicit published-definition and decision metadata instead of inferred legacy recipe state.


**Addendum — BE <-> FE Surface Contract**
- The primary redesign track is backend contract correction, not widget redesign. The critical outputs are: authoritative published definitions, fully compiled preview plans, generic execute-time plan running, explicit governance evaluation, and persisted explainability data.
- `EnvSync` remains the unchanged day-to-day operational widget. It should consume the new backend plan/run/explainability contract with minimal or no UX redesign.
- `EntityRegistry` remains a separate structural-definition workspace. It should evolve toward draft/validate/publish for entity definitions and runtime-affecting strategies rather than acting as a loosely authoritative live registry.
- `SyncAdmin` remains the single operational/governance console. It should be extended only where backend workflow justifies it and only in ways that simplify the admin mental model.
- `SyncProposals`, `SyncApprovals`, and `SyncEvidence` are not priority redesign targets by themselves. Their existing FE/BE implementations should be treated as reusable workflow surfaces that either become `SyncAdmin` sections or are retired later based on actual operational value.
- No FE surface should invent or infer business flow that is not already explicit in the backend plan/definition model. UI should render published definitions, compiled plans, gate decisions, and run evidence; it should not compensate for backend ambiguity.

**Addendum — Delivery Discipline**
- Deliver coherent end-to-end slices, not horizontal partial rewrites. Each slice should leave one complete boundary clearer and more correct than before.
- Recommended delivery order:
1. Definition and plan contract slice: authored definition -> published definition -> compiled preview plan types and persistence contract.
2. Preview/runtime slice: preview loads one published source, compiles explicit flow, and persists explainability/governance context.
3. Execute slice: execute becomes a plan runner with no contract-specific business branching outside the plan.
4. Governance slice: approvals/freeze/risk/environment references are evaluated and recorded consistently against the plan.
5. FE integration slice: repoint `EnvSync`, `EntityRegistry`, and `SyncAdmin` to the new backend contracts with minimal UI churn.
- For every slice, the acceptance test is the same: the backend source of truth is clearer, the runtime path is less ambiguous, and the existing widgets need less interpretation logic, not more.

**Implementation Status — FE Integration (Next Primary Slice)**

Approval-policy resolution and environment/service-policy publication are still partially external runtime policy surfaces rather than fully published-definition/package state. - `SyncAdmin` is structurally correct as the single governance/operations console, but it does not yet expose the new plan/explainability contract directly. It should remain the only admin console and grow only by adding backend-justified sections, not by becoming a second EnvSync.
